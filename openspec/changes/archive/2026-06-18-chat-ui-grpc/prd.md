# PRD：聊天畫面與 gRPC-Web 串流接通

- **狀態（triage）**：`ready-for-agent`

## Problem Statement

RAG 後端服務已搬進 Server 端，但尚未對外暴露，WebAssembly 前端無法使用——目前還沒有任何聊天畫面。開發者希望把來源專案那套已成熟的聊天體驗（逐字串流回應、工具呼叫提示、引用卡片、追問建議）帶進本專案，讓使用者能在瀏覽器與文件對話。

困難在於 render mode 落差：來源聊天頁是 server 端 Blazor，元件直接注入後端服務並在畫面端跑工具呼叫迴圈；本專案是 WebAssembly，畫面在瀏覽器執行，後端只在 Server，兩端必須透過 gRPC-Web 溝通。串流對話如何跨 gRPC-Web 傳遞、工具呼叫迴圈該在哪一側執行，是必須先解決的設計問題。

## Solution

在 Server 端建立一個聊天 gRPC 服務，把整個工具呼叫迴圈與串流對話都留在 Server：服務內部組好「載入文件」與「語意搜尋」兩個工具、呼叫既有的串流對話能力，把產生的事件（逐字文字、工具呼叫提示、回合狀態）以 server-streaming 串流給前端。前端只負責送出使用者輸入、接收事件串流並渲染——對應既有 Weather 範例服務的 gRPC-Web 串流範式。

追問建議以一次性 unary 呼叫接通。前端移轉來源那整套聊天畫面元件，視覺與互動儘可能與來源一致；本階段以「能運作、畫面一致」為先，樣式技術的統一留待後續重構。

## User Stories

1. 作為使用者，我想在聊天頁輸入問題並送出，以便就文件內容向助理提問。
2. 作為使用者，我想看到助理回應逐字出現（打字機效果），以便即時感知回應進度。
3. 作為使用者，我想在助理檢索時看到「載入文件」與「搜尋：<關鍵詞>」的提示卡片，以便理解它正在做什麼。
4. 作為使用者，我想在回應末尾看到引用卡片（檔名 + 摘錄），以便追溯答案來源。
5. 作為使用者，我想點擊引用卡片開啟對應的文件檢視器（Markdown 或 PDF），並跳到引用片段，以便查證原文。
6. 作為使用者，我想在一回合結束後看到最多三條追問建議按鈕，以便一鍵延續對話。
7. 作為使用者，我想按「New chat」清空對話重新開始，以便切換到新主題。
8. 作為使用者，我想在送出新訊息時，前一個尚在進行的回應被妥善取消，以便避免回應交錯。
9. 作為使用者，我想要助理的 Markdown 回應被正確渲染（標題、清單、強調），以便閱讀格式化內容。
10. 作為開發者，我想要工具呼叫迴圈完整留在 Server，以便前端無需持有金鑰或多次往返即可取得串流回應。
11. 作為開發者，我想要串流事件以扁平訊息加類別列舉表示，以便跨 gRPC-Web 穩定傳遞、避免多型序列化問題。
12. 作為開發者，我想要回合狀態（server 端對話 id）以串流事件回傳、前端存下後於下一回合帶回，以便串接多回合對話。
13. 作為開發者，我想要前端以具體 client 類別（非動態 proxy）呼叫聊天服務，以便相容 WebAssembly 沙盒限制。
14. 作為開發者，我想要追問建議走一次性 unary 呼叫，以便沿用既有 unary 範式、與串流分離。
15. 作為開發者，我想要既有 Weather／Counter 功能與既有 gRPC-Web 接線不受影響，以便不破壞現有能力。
16. 作為維護者，我想要前端的對話脈絡縮減（取最近數則訊息）留在前端，以便後端建議服務的輸入維持與來源一致、無需改動。

## Implementation Decisions

- **工具呼叫迴圈留在 Server**：Server 端新建聊天 gRPC 服務，內部組裝「載入文件」與「語意搜尋」兩個工具（呼叫 Server 的語意搜尋能力），執行既有串流對話流程；既有串流對話方法不改動。前端只渲染串出的事件。系統提示詞與建議提示詞隨工具編排一併移至 Server。
- **串流合約形狀**：server-streaming 方法。請求帶使用者輸入與上一回合狀態 id。串流訊息為**扁平結構 + 類別列舉**（逐字文字／工具呼叫提示／回合狀態三類），工具呼叫提示攜帶搜尋關鍵詞與檔名過濾。來源以回呼回傳的回合 id，改以一則「回合狀態」串流事件回傳。
- **追問建議**：獨立 unary 方法，請求帶已縮減的對話脈絡、回應帶建議字串集合。脈絡縮減邏輯留在前端。
- **前端 client**：具體 client 類別實作合約介面、以低階呼叫器發動 gRPC（WebAssembly 不支援動態 proxy），沿用既有 Weather client 的命名與序列化範式。
- **UI 元件移轉**：來源聊天頁與其子元件（標頭、訊息串、單則訊息、建議列、輸入框、引用卡片、載入動畫）連同各自的 scoped 樣式原樣搬入前端，兩種樣式技術暫時並存，優先確保運作。
- **第三方前端資產**：Markdown 渲染、消毒、Markdown／PDF 檢視器等前端函式庫一併搬入前端靜態資源，使畫面全功能；不搬來源的 CSS reset（與本專案既有樣式框架衝突）。
- **既有資產**：Weather 範例服務、其合約與 client 保留不動，新聊天服務與其並存。

## Testing Decisions

好的測試只驗外部行為，不綁實作細節。本案的可測接縫分兩層：

- **純邏輯單元測試**（自動）：
  - 串流事件扁平化映射——把內部串流事件轉成扁平合約訊息（含搜尋參數解析）的純轉換邏輯。
  - 前端對話脈絡縮減——取最近數則非空訊息的純整形邏輯。
  - 這兩者不依賴外部 OpenAI API，可獨立驗證。prior art：既有測試專案中對 token 切塊與向量庫的單元／round-trip 測試。
- **端到端 SIT**（手動）：送訊息→逐字串流→工具卡片→引用卡片→追問建議→New chat 的完整流程，需真實金鑰與瀏覽器。prior art：既有 Weather／Counter 的 SIT 流程。
- **不 mock OpenAI**：串流對話與建議服務依賴外部 API，不為其撰寫單元測試（以 SIT 涵蓋）。
- **回歸**：既有 Weather／Counter SIT 仍須通過。

## Out of Scope

- 把搬入的 scoped 樣式改寫為本專案的 utility 樣式框架（留待後續重構）。
- 改動既有串流對話、語意搜尋等後端服務的邏輯。
- 串流對話與建議服務的單元測試（依賴外部 API）。
- 既有 Weather 範例服務的任何變更。
- 文件擷取內容的擴充（沿用既有範例文件）。

## Further Notes

- 幾個實作中需釘的小決策：聊天頁掛在根路由或獨立路由、來源全域共用樣式的落點、本專案樣式框架的 reset 對 Markdown 預設樣式的影響——皆需實際執行後依視覺落差調整。
- 本案完成後，RAG 能力即從「Server 端可解析」進展為「前端可互動使用」。
